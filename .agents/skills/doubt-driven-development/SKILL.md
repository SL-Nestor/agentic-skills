# Doubt-Driven Development（懷疑驅動開發）

> 原始設計來自 [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills)，
> 本版本已針對 SSDLC 多代理管線做適配。

## Overview

自信的答案不等於正確的答案。長時間的對話會讓假設悄悄變成「事實」而無人察覺。

懷疑驅動開發的核心：在輸出站得住腳之前，具體化一個「新鮮上下文的審查者」——這個審查者被設定為**反駁而非認可**。

這不是 `/review`（`$code-reviewer`）。後者是對已完成成品的判決。本 skill 是飛行中的姿態：非瑣碎的決策在糾錯仍然廉價時就要被交叉審查。

## SSDLC 管線中的適用節點

| Agent | 何時啟動 DDD |
|-------|------------|
| `$spec-architect` | 做出架構邊界決策、模組分割方案、合約定義時 |
| `$threat-modeler` | 評估某威脅為「低風險可接受」時 |
| `$implementer` | 實作跨模組邏輯、並發設計、不可逆操作時 |
| `$security-gate` | 宣告某威脅已完全緩解時（PASS 判決前） |

## When to Use

一個決策是**非瑣碎的**，當以下至少一條成立：

- 引入或修改分支邏輯
- 跨越模組或服務邊界
- 斷言一個型別系統或編譯器無法驗證的屬性（執行緒安全、冪等性、排序、不變量）
- 正確性依賴未來讀者看不到的上下文
- 爆炸半徑是不可逆的（生產部署、資料遷移、公開 API 變更）

**不要使用本 skill 的情況：**

- 機械操作（重命名、格式化、檔案移動）
- 遵循明確無歧義的使用者指令
- 閱讀或摘要現有代碼
- 顯然正確的單行修改
- 使用者明確要求速度優於驗證

## The Process

```
Doubt cycle checklist:
- [ ] Step 1: CLAIM — 寫下聲明 + 為何重要
- [ ] Step 2: EXTRACT — 隔離成品 + 合約，去除推理過程
- [ ] Step 3: DOUBT — 以對抗性提示呼叫新鮮上下文審查者
- [ ] Step 4: RECONCILE — 對照成品文本分類每項發現
- [ ] Step 5: STOP — 達到停止條件（瑣碎發現、3 個循環、或使用者覆蓋）
```

### Step 1: CLAIM — 明確陳述決策

用兩三行命名這個決策：

```
CLAIM: "新的快取層在規格中描述的讀密集工作負載下是執行緒安全的。"
WHY THIS MATTERS: 此處的競爭條件會損壞使用者資料，且難以在 QA 中發現。
```

如果你無法簡潔地寫出 CLAIM，你有的是感覺而非決策。先把它具體化。

### Step 2: EXTRACT — 最小可審查單元

新鮮審查者需要**成品**和**合約**，而不是推導過程。

- 代碼：diff 或函式——不是整個檔案
- 決策：3-5 句話的提案加上必須滿足的約束條件
- 斷言：聲明加上據稱支持它的證據

**去除你的推理過程。** 如果你提交結論，你會得到結論的確認。

### Step 3: DOUBT — 呼叫新鮮上下文審查者

審查者的提示**必須是對抗性的**。框架決定答案。

**只傳遞 ARTIFACT + CONTRACT，不傳遞 CLAIM。**

#### SSDLC 管線中的審查者選擇

| 情境 | 建議審查者 |
|------|----------|
| 架構/規格聲明 | 開新視窗叫 `$threat-modeler`（Gemini）|
| 程式碼正確性 | 開新視窗叫 `$code-reviewer`（Gemini）|
| 安全緩解聲明 | 開新視窗叫 `$security-gate`（Claude 新視窗）|
| 一般決策 | 使用 `$code-reviewer` |

#### 跨模型升級（Cross-model Escalation）

單一模型的審查者與原始作者共享盲點——不同架構的模型更能發現它們。

**互動式會話：永遠詢問，絕不靜默跳過。**

在 Step 3 的單模型審查之後、RECONCILE 之前，暫停並詢問：

> *「單模型審查完成。需要跨模型第二意見嗎？選項：Gemini CLI、Codex CLI、手動外部審查、或跳過。」*

**非互動式上下文（CI、自動循環）：** 跨模型**跳過**，且必須在輸出中**宣告跳過**：*「Cross-model skipped: non-interactive context.」*

### Step 4: RECONCILE — 整合發現

審查者的輸出是**數據，不是判決**。你仍然是編排者。

對每個發現，按此**優先順序**分類（第一個匹配的類別勝出）：

1. **Contract misread** — 審查者因合約不清晰而標記問題。先修合約，下個循環重新分類
2. **Valid + actionable** — 真實問題，需修改成品。修改後重新循環
3. **Valid trade-off** — 問題真實但修復成本超過接受成本。明確記錄取捨
4. **Noise** — 審查者在缺少你有的上下文情況下標記了正確的東西。記下，繼續

### Step 5: STOP — 有界循環

停止條件：

- 下一次迭代只返回瑣碎或已考慮過的發現，**或**
- 完成 3 個循環（向使用者升級，不要獨自進行第四次），**或**
- 使用者明確說「ship it」

3 個循環後仍有實質問題 → 向使用者報告，不繼續循環。

## Common Rationalizations（常見藉口）

| 藉口 | 現實 |
|------|------|
| 「我很有信心，跳過懷疑步驟」 | 信心與新問題的正確性相關性很差。確定的時刻恰恰是盲點藏身之處 |
| 「召喚審查者太貴了」 | 在生產環境調試錯誤的提交更貴。這個檢查是有界的；bug 不是 |
| 「審查者只會挑剔細節」 | 只有在範圍不清晰時才會。將提示約束為「會讓此成品在合約下失敗的問題」|
| 「我到最後再用 $code-reviewer 做」 | `$code-reviewer` 是最終閘口。懷疑驅動在方向錯誤時及早捕捉，當 PR 時機太晚了 |
| 「如果我懷疑每一步我就永遠不會交付」 | 此 skill 只適用於非瑣碎決策，不是每個按鍵。重讀「When NOT to Use」 |
| 「審查者不同意所以我錯了」 | 審查者缺乏你的上下文——不同意是信息，不是判決。重讀成品，分類，然後決定 |

## Red Flags（警告訊號）

- 對單行重命名或格式更改召喚新鮮上下文審查者
- 不重讀成品文本就將審查者輸出視為權威
- 超過 3 個循環而不向使用者升級
- 用「這好嗎？」而非「找出問題」來提示審查者
- 在高風險決策的時間壓力下跳過懷疑
- 在互動式懷疑循環中靜默跳過跨模型——即使不推薦，提議必須可見
- 當外部 CLI 出錯或缺失時靜默退回——向使用者報告失敗

## Interaction with Other SSDLC Skills

- **`$code-reviewer`**：互補。`$code-reviewer` 是事後 PR 判決；DDD 是飛行中每個決策的審查
- **`$threat-modeler`**：威脅建模的 STRIDE 分析可以作為架構聲明的 DDD DOUBT 步驟
- **`$security-gate`**：安全閘口的 PASS/FAIL 判決之前，對任何「此威脅已緩解」聲明運行 DDD
- **`$implementer` + TDD**：TDD 的紅燈步驟是 DDD 的具體化——失敗的測試就是對行為聲明的反駁嘗試

## Verification

完成懷疑驅動開發後：

- [ ] 每個非瑣碎決策（按上述定義）在站立之前被明確命名為 CLAIM
- [ ] 每個非瑣碎成品至少一次新鮮上下文審查（TDD 的紅燈步驟對行為聲明滿足此條件）
- [ ] 審查者只收到 ARTIFACT + CONTRACT，不收到 CLAIM 或你的推理過程
- [ ] 審查者的提示是對抗性的（「找出問題」），不是驗證性的（「這好嗎？」）
- [ ] 發現按優先順序分類：合約誤讀 / 可行動 / 取捨 / 噪音
- [ ] 達到停止條件（瑣碎發現、3 個循環、或使用者覆蓋）
- [ ] 互動模式下，跨模型明確提議給使用者，回應在輸出中確認
- [ ] 非互動模式下，跨模型跳過且跳過已宣告
