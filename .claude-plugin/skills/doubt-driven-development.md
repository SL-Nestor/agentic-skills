# 🤔 懷疑驅動開發（Doubt-Driven Development）

> **Plugin Skill**: 載入此技能後，你將啟用 DDD 對抗性審查模式。
> 完整規範請參閱 `core/skills/doubt-driven-development/SKILL.md`。

## 概念

DDD 是一個 **對抗性決策審查框架**：  
讓新鮮上下文的審查者（另一個 AI 視角）挑戰每一個非瑣碎的決策，在問題固化前發現假設錯誤。

## 啟動方式

```
$ddd [聲明或決策]
```

## 五步驟工作流程

```
1. CLAIM      → 明確命名要審查的決策聲明
2. EXTRACT    → 從聲明中提取：合約 + 成品（不含推理）
3. DOUBT      → 新鮮上下文審查者嘗試反駁（「找出問題」）
4. RECONCILE  → 分類發現：合約誤讀 / 可行動 / 取捨 / 噪音
5. STOP       → 滿足停止條件時結束循環
```

## 停止條件

- 所有發現都是「噪音」或「可接受取捨」
- 已完成 3 個審查循環
- 使用者明確覆蓋（接受已知風險）

## 何時使用

| SSDLC Phase | 觸發時機 |
|-------------|---------|
| Phase 2（需求） | 對核心業務規則聲明 |
| Phase 3（架構） | 對技術選型決策 |
| Phase 5（實作） | 對複雜算法或安全實作 |
| Phase 7（安全閘口） | 對「此威脅已緩解」聲明 |

## 跨模型建議

理想情況下，審查者應使用不同 AI 模型（Gemini 審查 Claude 的聲明）。  
若無法切換模型，使用**新的對話視窗**作為替代方案。

## Verification

完成後確認：
- [ ] 審查者只收到 ARTIFACT + CONTRACT，未收到原始推理
- [ ] 審查者提示是對抗性的（「找出問題」），非驗證性的
- [ ] 達到停止條件後才結束循環
