# SSDLC v7.6: 工程管線自動化與左移 (Governance as Code)

## 核心目標
回應系統核可的《工程視角突圍建議》。將「文件化治理」轉型為能在日常開發管線中自動把關的「工程化治理」，包含依賴底層 SDK、指標左移與量化追蹤。

## 變更紀錄

### 1. 鋪裝路面哲學 (Paved Road & SDK-First)
- **變更點**：修改了 `ssdlc-core-rules.md`。
- **內容**：在核心架構限制中新增「Paved Road Principle」。強制 AI 在實作 Auth, Logging, Database 等底層功能時，**優先呼叫公司治理核可的 Shared SDK / Middleware**，而非自行硬刻原生邏輯。透過「利誘」達成合規。

### 2. 資料庫危機左移 (Shift-Left Data Defense)
- **變更點**：修改了 `lifecycle-plan/SKILL.md` 的 Enterprise Override。
- **內容**：原本的「無損破壞性遷移控制」通常在 Phase 4 或程式寫完後發生。現在強制**左移至 Phase 2 (Plan)**。只要發現本次任務會寫入/修改 DB Schema，AI 必須獨立開一個「Data Defense Task」，並向 DBA/架構師發出審查請求後，才能開始寫 Code。

### 3. 可觀測性防呆機制 (Telemetry Defaults)
- **變更點**：修改了 `lifecycle-build/SKILL.md`。
- **內容**：新增「Telemetry Defaults」。全面禁止 AI 使用單薄的 `logger.Info()`。強制要求任何 API/Web Request 都必須套用 Middleware 來進行 `TraceId` 與 `CorrelationId` 的攔截與轉發。將可觀測性的負擔轉移給底層架構。

### 4. 治理代價的量化 (DORA Metrics Telemetry)
- **變更點**：修改了 `ssdlc-tracker-template.md`。
- **內容**：在所有的 Tracker 尾端新增了 **DORA Metrics Telemetry** 區塊。強制紀錄「需求交接時刻」、「實際開發啟動時刻」、「PR 建立時刻」以及最重要的「**DORA Blocked Duration (被 Writeback 退回而卡住的時間)**」。這讓管理層能清楚看見前置時間 (Lead Time) 被拖慢的真正瓶頸在哪裡。

## 結論
v7.6 正式讓 SSDLC 具備了「SRE 維運意識」。它不再只防禦 Bug，也防禦治理流程帶來的管理風險。用量化數據與鋪裝路面的方式，實現真正的技術突圍。
