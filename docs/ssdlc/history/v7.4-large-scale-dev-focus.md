# SSDLC v7.4: 大型軟體工程視角 (Large-Scale Dev Focus)

## 核心目標
解決第三輪深度審計中發現的「大型協作缺口」，將 SSDLC 從敏捷工具正式升格為能支撐百人團隊、微服務維運的企業級開發引擎。

## 背景
經過 v7.1 ~ v7.3 的反覆除錯與治理對接，SSDLC 已經具備了強大的規範約束力。但在真實的高可用性 (High Availability) 和持續整合 (CI/CD) 環境下，舊版仍存在四個致命傷：
1. **Prompt 肥大症**：27 個 Omni-Skill 巨集強制塞在主文件，降低了 AI 的執行精準度，也與公司級 `skill-registry` (GOV-012) 發生權威重疊。
2. **缺乏 CI 邊界**：系統預設開發完就「直接 committed」，這在真實大廠會立刻被 Branch Protection 阻擋。
3. **缺乏退路**：沒有在計畫階段要求 Feature Flag 與 Rollback 機制。
4. **維運盲區**：可觀測性 (Observability) 只是「指引」，卻未曾在最後一關被強制驗證。

## 變更紀錄

### 1. Token 最佳化與權威移交 (Prompt Bloat Reduction)
- **修正 `copilot-instructions.md`**：移除了 22 個非核心的 Omni-skill (如 UI/UX、DB、DevOps 等)。
- **引入 GOV-012 指標**：加入 `Governance Notice`，明確指示若需要領域專用技能，必須去 `TwReuse/sl_company-platform-governance/docs/governance/skill-registry.md` 查詢，消除兩套標準的衝突。

### 2. 交付邊界對齊 (Pull Request Boundary)
- **修正 `copilot-instructions.md` 與 `lifecycle-ship/SKILL.md`**：
  - 放棄 `VCS Checkpoint committed` 的描述。
  - 將最終的交付標準改為：**Feature Branch Pushed & Pull Request Draft created**。
  - 確保開發者的最後一步是交接給 CI/CD Pipeline 與人類 Reviewer。

### 3. 可維運性強制化 (Operability Enforcement)
- **修正 `lifecycle-plan/SKILL.md`**：
  - 在 Enterprise Override 與驗證條件中加入：必須制定 **功能開關 (Feature Flag) 策略** 以及明確的 **退版計畫 (Rollback Plan)**。將「部署」與「發布」解耦。
- **修正 `lifecycle-verify/SKILL.md`**：
  - 在驗證條件中明確加入 **可觀測性驗證 (Observability validated)**，確保 Tracing、Logging 與 Metrics 有被實際實作出來，防止盲目上線。

## 結論
v7.4 的 SSDLC 正式移除了「玩具/單兵專案」的刻板印象。現在的流程不僅能產出程式碼，還能確保產出符合 PR/CI 規範、帶有退路機制，且為 Day 2 維運做好了遙測準備。這是邁向大規模 AI 軟體工程的最後一塊拼圖。
