# REQ (需求端) 到 ENG (工程端) 交付對照與差異指引

本文件專為工程師與自動化代理人 (AI Agents) 撰寫，旨在釐清「公司高階治理架構 (Governance)」與「SSDLC 開發加速器」兩者在工作現場的邊界與合作模式。

---

## 1. 核心觀念：法律與執法的關係

請不要將 SSDLC 與公司治理範本視為「二選一」的互斥產品。
*   **公司治理 (Governance, 包含 tpl_*.md)**：是系統開發的「四倉邊界法律」。它定義了文件要怎麼傳遞、合約誰負責、以及交付包的標準。
*   **SSDLC Autopilot (位於 `_inbox/` 或開發用腳本)**：是具備高效率驗證能力的「執法機器人」。它是**工程師參考對照用的工具**，用來幫你高速遵守上述法律。它**不是** SSOT (單一真相來源)。

## 2. 關於「唯一真相」的思維轉換 (SSOT)

在使用 SSDLC 時，很多人會被「規格輸入檢查單」誤導，以為 PRD 就是一切。在公司治理下，請嚴守以下紅線：

| 維度 | SSDLC 視角 (防呆檢查清單) | 公司治理 (正式合約) | 工程師實作指南 |
| :--- | :--- | :--- | :--- |
| **技術棧宣告** | 要求在輸入中提供，以免 AI 盲目瞎猜。 | 認為不該污染 PRD，應由開發端自行管理。 | **實作法**：工程師應在交付包的 metadata 或 `ENG README` 中寫明，啟動 AI 時當作參數傳入，而不是逼 PM 寫進 PRD。 |
| **API 欄位與結構** | 依賴 Spec 中萃取，或是動態推演。 | **OpenAPI / JSON Schema 才是王** (綁定 `baseline_ref`)。 | **實作法**：AI 若覺得架構需要新欄位，**嚴禁**直接改程式碼！必須走公司 GOV-004 的 **Writeback (回寫法則)** 修改 OpenAPI 契約並等候合併。 |
| **啟動閘門** | 依賴 `/start-ssdlc` 與 `Gate P/A/B` 等名詞。 | 依賴 GOV-006 的 `phase-1/phase-2` 追溯基準。 | **實作法**：SSDLC 內的 Gate 只是 AI 的「內部思考暫停點」，專案放行依舊以公司治理狀態為準。 |

## 3. 開倉與多模組 (Monorepo) 管理約定

對於擁有 20+ 模組的大型專案（如：e首發票）：
1. **開倉方式 (B+A 架構)**：請使用 GitHub 的「公司範本倉」 (B) 先複製出統一骨架，開倉後再使用腳本 (A) 自動替換 `module_id` 與對齊要求的 PRD 連結。不要每次都讓 AI 從零捏造資料夾結構。
2. **架構方針 (Monorepo)**：我們實行 **「一系統一工程倉」**，不同模組統一放在同一個大 Repository 下（如 `src/modules/LOGIN/`）。
3. **防碰撞策略 (Version Consistency)**：多個工程師與多個 AI 不會產生版本分裂的終極保證是：**「大家都讀同一份 OpenAPI Baseline」**。遵守了 Writeback 條款，就不會發生架構衝突。

## 4. 雙軌制開發 (何時該用什麼模式？)

不用擔心加入治理後，連個小腳本都寫不動了。這取決於你如何「發動」你的開發流程：

*   🚀 **敏捷模式 (日常工具 / 獨立腳本 / Prototype)**：如果你只是想做一個不需要跨部門合約的小工具，請直接給 AI 你的 PRD 讓它自由發揮。
*   🛡️ **企業模式 (e首發票 / 跨模組核心)**：如果你要動核心系統，請務必按照 `tpl_req_eng_handoff_checklist.md` 準備好「交接包裹」，明確指定 OpenAPI Baseline 讓 AI 跟隨，嚴格控制邊界。
